## Разработка локальной системы автоматического составления технической документации по аудиозаписям обсуждений

**Цель проекта:** Разработать безопасную, автономную (работающую в локальной сети/на локальном сервере) и удобную систему, которая автоматически транскрибирует аудиозаписи технических совещаний, анализирует их с помощью LLM и генерирует структурированную техническую документацию (спецификации, протоколы встреч, списки задач).

### 1. Анализ требований и проектирование архитектуры

#### 1.1. Определение спецификаций
*   **Анализ источников аудио:** Определение поддерживаемых форматов (MP3, WAV, M4A), источников (файловое хранилище, сетевая папка, загрузка через веб-интерфейс, микрофоны в переговорной).
*   **Определение типов выходных документов:**
    *   **Протокол встречи (Meeting Minutes):** Краткое содержание, список участников, ключевые решения.
    *   **Техническое задание (ТЗ/Specification):** Извлечение функциональных и нефункциональных требований из обсуждения.
    *   **Бэклог задач (Task List):** Автоматическое создание задач в формате "Что сделать?" с пометкой ответственного (если упоминался).
    *   **Словарь терминов:** Выделение специфических терминов и их определений из контекста.
*   **Сбор требований к точности:** Определение необходимости дикторской разметки (кто что сказал) и уровня детализации.

#### 1.2. Выбор технологий и системный дизайн
*   **Выбор стека технологий:**
    *   **Бэкенд:** Python (FastAPI / Django) — лучший выбор для работы с ML/AI.
    *   **Фронтенд:** React / Vue.js для удобного интерфейса.
    *   **База данных:** PostgreSQL для хранения метаданных, задач и статусов. Можно использовать векторную БД (Qdrant, Chroma) для хранения эмбеддингов семантического поиска по документам.
    *   **Очереди задач:** Celery + Redis для обработки длительных задач (транскрибация и генерация документации).
*   **Проектирование схемы взаимодействия:**
    `Пользователь (Web UI) ↔ Веб-сервер ↔ Очередь задач (Celery) ↔ Обработчики (ASR + LLM) ↔ Файловое хранилище`
*   **Проектирование структуры БД:** Хранение информации о записях, статусах обработки, ссылках на сгенерированные документы и аудиофайлы.
*   **Выбор моделей машинного обучения:**
    *   **ASR (Распознавание речи):** Выбор локальной модели (например, `Whisper` от OpenAI в вариантах `base/large`, `nemo`, `Vosk`) с упором на качество распознавания технической лексики и скорость работы.
    *   **LLM (Генерация документации):** Выбор локальной модели (например, `Llama 3`, `Mistral`, `Qwen` в квантованных версиях) для работы без доступа в интернет. Тонкая настройка системного промпта для получения структурированного вывода (JSON/Markdown).

### 2. Разработка бэкенда и ядра обработки

#### 2.1. Разработка ядра веб-сервера
*   Реализация API для загрузки аудиофайлов, управления задачами на обработку, получения статусов транскрибации.
*   Разработка механизмов аутентификации и авторизации (ролевой доступ: администратор, инженер, менеджер).
*   Создание системы управления промптами для LLM (возможность редактировать шаблоны генерации документации).

#### 2.2. Разработка "Агентов" обработки (Обработчиков)
*   Разработка модуля **Transcriber Service** (Сервис транскрибации):
    *   Загрузка аудиофайла из очереди.
    *   Запуск локальной ASR-модели для преобразования речи в текст (с таймкодами).
    *   Сохранение "сырой" транскрипции в БД.
*   Разработка модуля **Document Generator Service** (Сервис генерации документации):
    *   Получение транскрипции из очереди после завершения ASR.
    *   Сегментация текста на логические блоки (если диалог длинный).
    *   Взаимодействие с локальной LLM через API (например, `llama.cpp`, `vLLM`, `Ollama`).
    *   Генерация структурированных документов на основе настроенных промптов.
    *   Обеспечение безопасной работы в изолированной среде.

### 3. Разработка клиентского веб-интерфейса (Фронтенд)

#### 3.1. Разработка панели управления
*   Дизайн интуитивного интерфейса для загрузки файлов (drag-and-drop).
*   Реализация дашборда со списком всех обработанных записей и статусами (Ожидание / Транскрибация / Генерация / Готово / Ошибка).
*   Разработка компонентов для визуализации аудио (волны) с синхронизированным текстом транскрипции.

#### 3.2. Интерфейс просмотра и редактирования документации
*   **Редактор транскрипции:** Возможность просматривать текст, исправлять ошибки распознавания (если есть), просматривать таймкоды.
*   **Редактор документа:** Отображение сгенерированного документа (Markdown/HTML). Пользователь может скачать результат в нужном формате (PDF, .docx, .txt) или отредактировать его прямо в браузере перед сохранением.
*   **Боковая панель "Ключевые решения/Задачи":** Автоматически извлеченный список задач и решений с возможностью утверждения.

### 4. Интеграция системы и настройка LLM

#### 4.1. Системная интеграция
*   Связывание всех компонентов (API Gateway, Worker'ы, БД, Хранилище) в единый docker-compose стек для легкого развертывания.
*   Сквозное тестирование полного цикла: загрузка файла → постановка в очередь → транскрибация → генерация документа → отображение в UI.
*   Настройка уведомлений (email/Telegram) о завершении обработки длинных файлов.

#### 4.2. Реализация механизмов генерации документации (RAG)
*   Разработка или адаптация промптов для извлечения сути технических обсуждений.
*   Настройка взаимодействия LLM с "внешним инструментом" (например, для поиска определений терминов в корпоративной вики), если система будет расширяться до RAG (Retrieval-Augmented Generation).

### 5. Тестирование, документация и развертывание

#### 5.1. Комплексное тестирование
*   **Тестирование безопасности:** Проверка, что система не отправляет данные во внешние API (если это запрещено), защита от несанкционированного доступа к API.
*   **Нагрузочное тестирование:** Проверка стабильности работы при одновременной обработке нескольких длинных записей.
*   **Тестирование отказоустойчивости:** Поведение системы при падении одного из сервисов (например, LLM), возможность перезапуска задач.
*   **Оценка качества:** Сравнение сгенерированных документов с "эталонными" ручными протоколами (метрики ROUGE, BLEU, но с упором на фактическую точность).

#### 5.2. Завершение проекта
*   Подготовка пользовательской документации (инструкция для инженеров) и технической документации (для администраторов системы).
*   Обучение персонала работе с новой системой.
*   Развертывание финальной версии системы на выделенном локальном сервере или мощной рабочей станции.
*   Планирование следующих этапов: добавление поддержки видео, автоматический перевод документации, интеграция с Jira/YouTrack для создания задач.
